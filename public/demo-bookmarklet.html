2 · Use it

  1. (For live editing) run npm run vite in ~/ello-storefront-app. It serves the widget from http://localhost:3000. If it's not running, the bookmarklet falls back to the deployed Cloud Run widget automatically.
  2. Open a prospect's product page (e.g. madhappy.com or youngla.com → any product).
  3. Click the Ello Try-On Demo bookmark. A native Try it on button appears by Add-to-Cart, plus the floating launcher. Click it, upload a photo once, and the try-on renders on the prospect's actual product image.
Settings gear ⚙ (bottom-left): change the demo live, just like the dashboard. Switch Result shows in → Popup widget vs PDP image (turns the widget off and drops the try-on onto the product photo), toggle the floating launcher / inline button / Complete the Look, and set the brand color + button text. Changes apply instantly and are remembered for next time.
Console controls: window.__elloDemo.tryOn() (fire a try-on), .open() (fitting room), .set('mode','mirror') (change a setting), .settings() (see current), .ctx() (product data read off the page), .reset().

What it wires up

Config + authResolves to ello-dev-store (its slug/token/plan). Try-ons POST storeSlug=ello-dev-store, which is all the /tryon proxy needs — no per-store secret.
Product imageRead live from the prospect PDP (ShopifyAnalytics.meta + /products/<handle>.js for the selected variant), so you try on their product.
Complete the LookCurated Search & Discovery pairings don't exist on a prospect store, so it classifies the tried-on item (top / bottom / dress / outerwear / footwear / accessory) and picks a complementary category from the store's own catalog — a top pairs with a bottom, a dress with a jacket or shoes — instead of another of the same thing. Demo-only.
SafetySets window.__ELLO_DEMO__; the un-curated CTL fallback is gated on it, and the local origin is localhost-only — real shoppers are never affected.

Local vs Cloud (important)

Many big Shopify stores (incl. youngla.com and madhappy.com) send a upgrade-insecure-requests / block-all-mixed-content CSP. That upgrades http://localhost:3000 to https://localhost:3000, which fails — so on those stores the widget loads from Cloud Run over https instead (the bookmarklet auto-falls back after ~0.7s). The demo still works; you just won't see un-deployed local edits.

Live local editsWork on stores without that CSP. To get them on a CSP store, run vite over HTTPS with a locally-trusted cert (mkcert) so https://localhost:3000 resolves, or just deploy.
CSP stores (youngla/madhappy)Deploy demo-bookmarklet.js + widget-main.js to the public Cloud Run service first; then the bookmarklet loads everything over https and works natively on their PDP.

Troubleshooting

Nothing happened? Open DevTools console. If you see the "could not load" alert, either npm run vite isn't running and the Cloud Run copy isn't deployed yet, or the store's CSP blocked both. Deploy to Cloud Run to cover CSP stores.
Launcher not showing? The inline button only injects on /products/ pages; elsewhere use the floating launcher or window.__elloDemo.open(). The ello-dev-store has a per-shopper try-on cap (~15/day) — heavy repeat testing from one IP can 429.
